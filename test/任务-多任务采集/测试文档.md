# 多任务采集测试文档

## 测试场景描述

### 角色：甲方IT经理

### 测试目的
验证系统多任务并行执行能力，特别是TCP服务端、TCP客户端和业务逻辑任务的协同工作。

## 任务配置

### 任务1：TCP服务端
- **类型**：监听事务类任务
- **配置**：
  - 端口：8888
  - 监听模式：广播模式
  - 编码格式：16进制

### 任务2：TCP客户端（摄像头通信）
- **类型**：定时任务
- **配置**：
  - 目标IP：192.168.1.100
  - 端口：9999
  - 编码格式：16进制
  - 轮询间隔：10秒

### 任务3：业务逻辑执行
- **类型**：流程任务
- **配置**：按照以下8个步骤执行

## 测试流程

### 步骤1：获取摄像头参数
1. 通过TCP客户端向摄像头发送获取参数指令
2. 接收返回的16进制数据
3. 将16进制数据转换为10进制
4. 解析为数组格式

### 步骤2：数据验证与存储
1. 将解析后的数组存储到临时变量 `rs`
2. 验证数组长度是否大于6
3. 如果长度不大于6，终止流程
4. 如果长度大于6，继续执行下一步

### 步骤3：广播数据
1. 将6组数据加工为固定JSON格式：
   ```json
   {
     "type": "camera_data",
     "data": [v1, v2, v3, v4, v5, v6]
   }
   ```
2. 将JSON转换为16进制
3. 通过TCP服务端广播给所有连接的客户端

### 步骤4：监听指令
1. 监听TCP服务端接收到的数据
2. 判断是否为指定指令或截取后等于指定指令
3. 等待指令到达（监听模式）

### 步骤5：指令验证
1. 如果接收到指定指令，继续执行
2. 如果未接收到指定指令，持续等待

### 步骤6：数据转换与传输
1. 将步骤2中的数组转换为 v1,v2,v3 模式：
   ```json
   {
     "v1": value1,
     "v2": value2,
     "v3": value3
   }
   ```
2. 将转换后的数据转为16进制
3. 通过TCP客户端发送给另一个设备
4. 等待设备返回数据
5. 判断返回数据是否包含固定前缀
6. 如果包含前缀，执行数据转换：去掉前缀并16转10进制
7. 解析为JSON格式

### 步骤7：数据存储
1. 将解析后的数据存储到日志文件
2. 将数据存储到数据记录系统

### 步骤8：HTTP推送
1. 将记录的数据通过GET请求推送到指定HTTP接口
2. 传递相关变量参数

## 预期结果

1. **任务启动**：三个任务成功启动并运行
2. **数据获取**：成功从摄像头获取参数并解析
3. **数据验证**：正确验证数组长度
4. **数据广播**：TCP服务端成功广播数据
5. **指令监听**：正确接收到指定指令
6. **数据传输**：成功与另一设备通信并接收响应
7. **数据存储**：数据正确存储到日志和记录系统
8. **HTTP推送**：成功推送数据到HTTP接口
