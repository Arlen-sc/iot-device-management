# 流程引擎日志系统设计与优化方案

## 1. 当前日志逻辑梳理

目前系统中的流程日志分为两层：

### 1.1 内存实时日志 (In-Memory Log)
- **存储位置**：`FlowExecutionContext.executionLog` (目前为 `List<String>`)
- **生命周期**：伴随单次流程执行的生命周期。执行结束后，直接作为 API 响应返回给前端。
- **记录内容**：
  - 流程的开始与结束。
  - 每个节点执行前后的上下文变量快照。
  - 节点执行的流转路径。
  - 节点执行成功、警告或异常信息。
- **展示位置**：前端的“流程调试控制台” (Debug Console) 以及单次执行的“执行结果”弹窗。

### 1.2 数据库持久化日志 (Persistent Log)
- **存储位置**：SQLite 数据库的 `flow_execution_log` 表。
- **生命周期**：长期持久化存储。
- **记录时机**：
  - **显式记录**：当流程中执行到专门的 `LOG` (日志记录) 节点时，会将用户配置的信息和数据落库。
  - **异常兜底**：当任意节点执行抛出异常或返回失败结果时，`FlowExecutor` 会自动捕获并生成一条 `ERROR` 级别的日志落库。
- **展示位置**：任务列表点击“日志”按钮弹出的历史日志界面。

## 2. 多角色评判 (Role-based Evaluation)

### 2.1 开发者 (Developer / Node Implementer)
- **诉求**：在调试节点逻辑或开发新节点时，需要清晰地看到每个节点的**耗时**、**输入输出数据结构**、以及**明确的错误堆栈**。
- **痛点**：目前内存日志是将所有信息拼成了一个长字符串 (`[时间] [类型] [节点] 消息`)，前端再用正则去解析。这种方式非常脆弱，且如果数据中包含换行符或特殊字符容易解析失败。同时缺乏节点的执行耗时统计。
- **优化方向**：**将内存日志结构化**。使用对象 (如 `LogEntry`) 存储日志，前端直接通过 JSON 字段渲染。增加节点执行耗时统计。

### 2.2 运维与管理员 (Ops / Admin)
- **诉求**：需要排查历史任务的运行情况，统计失败率，或者追溯某个特定设备的控制记录。
- **痛点**：历史日志只记录了 ERROR 和显式的 LOG 节点。如果任务正常结束但逻辑不符合预期，运维无法从历史日志中看出数据流转过程。
- **优化方向**：由于系统主要为轻量级 IoT 平台，全量落库会导致 SQLite 数据库急剧膨胀，保持当前的“异常自动落库 + 关键节点手动落库”是合理的。但应该在任务的级别上补充“最近一次执行日志”的缓存机制，以兼顾性能和追溯。

### 2.3 业务用户 / 流程设计者 (End User)
- **诉求**：在设计器中调试时，希望能直观地看出数据是经过哪些节点变换的，哪个节点抛出了什么业务错误。
- **痛点**：目前调试界面的日志像传统的控制台黑框，文本混杂在一起，阅读大段 JSON 变量时非常困难。
- **优化方向**：调试界面的日志展示应彻底告别文本解析，改为结构化的渲染，明确标识耗时、节点名，分离消息文本与数据。

---

## 3. 优化实施计划 (实施中)

基于以上评判，我们将进行以下代码层面的优化：

1. **结构化内存日志 (后端)**：
   - 新增 `ExecutionLogEntry` 类，包含 `timestamp`, `level`, `actionType`, `nodeName`, `message`, `data`, `durationMs` 等明确字段。
   - 修改 `FlowExecutionContext` 的 `executionLog` 存储结构为 `List<ExecutionLogEntry>`。
2. **记录节点耗时 (后端)**：
   - 在 `FlowExecutor` 中记录节点开始与结束的系统时间，将耗时计算结果随成功/失败日志一并写入。
3. **前端渲染改造 (前端)**：
   - 重构 `debug-console.js` 和 `task-list.js` 中的 `showExecResult` 方法。
   - 移除脆弱的正则表达式解析，直接读取日志对象的属性进行精准排版。
   - 增加执行耗时 (如 `12ms`) 的徽章展示，提升专业感。
